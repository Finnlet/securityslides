The tale of the new site.

Harpers's LLC, a small development in Alabama, started work on a new website

The site (at a brand new domain) was provisioned 48 hours before it was hacked.

The site was in maintenance mode

Background

A new domain was ordered and a site setup on that domain name.

The site had all up to date modules installed and an up to date core

The site as going to be a brochure type site with only one admin account. Think a basic corporate site.

This was setup on a new development server, dedicated to dev sites for Harpers's LLC

On the 3rd day of development

Jeremy, the content manager, was going to setup content types and notice the site had porn all over the front page

Jeremy was not happy

What?

Jeremy the in house IT person,Jean, who started looking at logs to see how an up to date Drupal site got compromised.

Jean pulled the access logs for the server and found only known trusted IP's accessed the site.

Jean had no idea how it happened so she restored a backup

2 days later

More porn spam

Jeremy lost all of his work.

Jean searched the logs again

Found only the same trusted IP's

Only computers inside of Harpers's LLC offices had accessed the site

Jean Ran a virus scan on the computers that had accessed the site

The scan was clean

Jean had no idea what was going on....

Any Ideas?

Is there a ghost?

Apache single user

By default apache on most OS's will run as a single user

That user has access to READ all the files for all sites on the system

In some cases that user has access to write to all files on the system. (This is bad)

What happened

Someone had compromised another site on the server.

Using that site, they were able to read settings.php and connect to the new Drupal site's database

They were able to inject spam by updating the table directly

The script used search the entire file system for settings.php and then ran commands to update the database on any site it found.

Lessons Learned

Do not allow a single apache instance to access multiple sites. Configure privilege separation.

Apache config is outside the scope of this presentation, but look into suphp, php-fpm and mod-itk

Be-careful of using shared hosting without proper configuration

Don't bring a site back online after a compromise unless you understand how and prevented future compromises
